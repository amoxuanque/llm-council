# LLM Council 代码质量与落地性分析报告

## 一、项目概述

LLM Council 是一个三阶段多模型协作系统：**独立回答 → 匿名互评 → 主席综合**。通过匿名化评审防止模型间偏见，通过多模型投票提升回答质量。

## 二、架构亮点

| 特性 | 实现方式 |
|------|----------|
| 并行查询 | `asyncio.gather()` 并发调用所有模型，单模型失败不阻塞流程 |
| 匿名互评 | Stage 2 将回答匿名为 Response A/B/C，模型自动排除自己的回答 |
| 流式推送 | SSE 逐阶段推送，含跨 chunk 行缓冲，前端实时展示进度 |
| 并发安全 | `fcntl.flock` 文件锁保护 read-modify-write 操作 |
| 数据持久化 | metadata（label_to_model、aggregate_rankings）随对话一起存储，刷新不丢失 |
| 推理模型适配 | 正确提取 DeepSeek-R1 的 `reasoning_content`，包裹 `<thinking>` 标签供评审参考 |
| 排名解析容错 | 支持英文 "FINAL RANKING:" / 中文 "最终排名："、"回答A" 等多种格式 |
| 前端安全 | 状态更新深拷贝避免 mutation bug；正则特殊字符转义防注入 |

## 三、当前模型配置

| 角色 | 模型 | 说明 |
|------|------|------|
| 议员 | DeepSeek-R1 | 推理模型，输出思维链 |
| 议员 | Qwen3.5-397B-A17B | MoE 大模型 |
| 议员 | Kimi-K2-Instruct | 月之暗面 |
| 议员 | GLM-4.6 | 智谱 AI |
| 主席 | DeepSeek-V3.2 | 独立于议会，避免利益冲突 |

主席模型不参与 Stage 1 投票，确保 Stage 3 综合的公正性。四个议员来自不同厂商/架构，最大化观点多样性。

## 四、剩余优化方向

### 工程层面

| 方向 | 说明 |
|------|------|
| Error Boundary | 前端缺少 React Error Boundary，组件异常会导致白屏 |
| 数据库迁移 | JSON 文件存储适合小规模，高并发场景应考虑 SQLite/PostgreSQL |
| httpx 客户端生命周期 | 当前模块级全局变量，多 worker 部署建议改用 FastAPI lifespan 管理 |
| SSE 异常恢复 | Stage 1 成功但 Stage 2 失败时，对话状态可能不一致，需增加回滚或补偿逻辑 |

### 产品层面

| 方向 | 说明 |
|------|------|
| 可配置议会 | 通过 UI 动态选择模型，而非硬编码 |
| 对话导出 | 支持 Markdown/PDF 导出 |
| 模型表现统计 | 长期追踪各模型的排名趋势 |
| 自定义评审标准 | 支持按准确性、创造性等维度分别评分 |

## 五、总结

项目已完成从概念验证到可用状态的升级。核心架构（匿名互评 + 主席综合）设计合理，关键工程问题（并发安全、数据持久化、SSE 解析、状态管理、正则安全）均已解决。当前可直接用于小规模部署，后续可根据使用规模渐进优化存储层和部署架构。
